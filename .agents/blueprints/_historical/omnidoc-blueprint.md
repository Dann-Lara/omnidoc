## 1. Project Overview

### Vision

OmniDoc es una plataforma SaaS multitenant diseñada para transformar la gestión de consultas médicas. Permite a médicos individuales y clínicas/hospitales optimizar sus operaciones con herramientas asistidas por IA, desde agenda inteligente hasta análisis predictivo. El sistema soporta múltiples niveles de usuario: Superadmin SaaS, Operadores de soporte, y Tenants con sus propios usuarios subordinados y permisos granulares.

### Goals

- Plataforma SaaS multitenant con aislamiento completo de datos entre clientes
- Dashboard de analytics con IA para decisiones clínicas optimizadas
- Módulo de citas médicas inteligente con predicción de estados
- Sistema de suscripciones con Stripe y gates de features dinámicos
- AI Core con fallback automático entre agentes y soporte para API keys de usuario
- Cumplimiento HIPAA-ready para integridad de datos médicos

### Success Metrics

- Tiempo de agenda de citas reducido en 60% mediante predicción IA
- Onboarding de nuevo tenant < 5 minutos
- 99.9% uptime del sistema
- Time-to-first-cita < 10 minutos post-registro

### Key Endpoints Detail

**POST /api/tenants**
```typescript
// Request
{
  name: string,
  type: "INDIVIDUAL" | "CLINIC",
  specialty?: string,
  willHaveSubordinates?: boolean,
  branding?: { logo?: string, colors?: { primary: string, secondary: string } },
  owner: {
    email: string,
    firstName: string,
    lastName: string
  }

// Response 201
{
  id: string,
  slug: string,
  status: "pending_onboarding",
  suggestedPlan: string

```

**POST /api/appointments/predict**
```typescript
// Request
{
  doctorId: string,
  dateRange: { start: Date, end: Date },
  patientHistory?: string[]
}

// Response
{
  predictions: [
    {
      date: Date,
      predictedStatus: "confirmed" | "no_show" | "canceled",
      confidence: number,
      factors: string[]
    }
  ],
  recommendations: string[]
}
```

**POST /api/webhooks/stripe**
- Handles: `customer.subscription.created`, `customer.subscription.updated`, `customer.subscription.deleted`, `invoice.payment_succeeded`, `invoice.payment_failed`

- Sends email notifications for payment events

### Component Style

- **Aesthetic:** "Clinical Architect" — editorial, authoritative, data-driven
- **No 1px borders** for sectioning — use tonal shifts (surface vs surface-container-low)
- **Glassmorphism:** 80% opacity + backdrop-blur(20px) para elementos flotantes
- **Primary buttons:** Gradient #00355f → #0f4c81 (135deg angle)
- **Inputs:** Fill-style con surface-container-high, focus indicator pill a la izquierda
- **Cards:** No dividers — spacing between items only
- **Zebra stripes:** Alternar surface vs surface-container para filas
- **Ambient shadows:** 40px blur, 0 offset, 6% opacity de on-surface

## 8. Authentication & Authorization

### Auth Flow (Supabase Auth - CORREGIDO)

```
================================================================================
 FLUJO COMPLETO DE AUTENTICACIÓN
================================================================================

───────────────────────────────────────────────────────────────────────────────
 SIGNUP (Nuevo Cliente)
───────────────────────────────────────────────────────────────────────────────

LANDING ───► /signup ───► POST /auth/signup ───► Supabase Auth
                                                          │
                        ┌─────────────────────────┼─────────────────────────┐
                        │                         │                         │
                  Email confirmado            Email NO confirmado           Error
                        │                         │                         │
                        ▼                         ▼                         ▼
                  Redirigir a            Mostrar pantalla             Mostrar error
                  /{org_slug}/dashboard    "Revisa tu email"           en formulario
                                             │
                                             └── Link para reenviar

───────────────────────────────────────────────────────────────────────────────
 LOGIN
───────────────────────────────────────────────────────────────────────────────

/login ───► POST /auth/login ───► Supabase Auth
                                        │
                                        ▼
                              ┌─────────────────┐
                              │ Login exitoso    │
                              └────────┬────────┘
                                       │
                                       ▼
                            ┌─────────────────────────┐
                            │ Check rol del usuario    │
                            └────────┬────────────────┘
                                     │
        ┌─────────────────────┬──────┴──────┬─────────────────────┐
        │                     │              │                     │
   SUPERADMIN/OPERATOR      CLIENT          CLIENT            SUBORDINATE
        │                     │ (con org)     │ (sin org)          │
        ▼                     ▼               ▼                     ▼
   /admin           /{org_slug}/dashboard   /{first_name}/dashboard  /{tenant_org_slug}/dashboard

───────────────────────────────────────────────────────────────────────────────
 FORGOT PASSWORD (Recuperar contraseña)
───────────────────────────────────────────────────────────────────────────────

/login ───► [¿Olvidaste tu contraseña?] ───► /forgot-password
                                                           │
                                                           ▼
                                                   POST /auth/forgot-password
                                                           │
                                                           ▼
                                               Supabase resetPasswordForEmail()
                                                           │
                                                           ▼
                                                   Mostrar "Revisa tu email"
                                                   (siempre, por seguridad)

───────────────────────────────────────────────────────────────────────────────
 RESET PASSWORD (Desde link del email)
───────────────────────────────────────────────────────────────────────────────

Email link ───► /reset-password?token=xxx ───► POST /auth/reset-password
                                                              │
                                                              ▼
                                                     Supabase updateUser()
                                                              │
                                                              ▼
                                                     Redirect a /login
                                                     (con mensaje de éxito)

================================================================================
 SISTEMA DE INVITACIONES (Token-based)
================================================================================

Superadmin crea Operator:
   ├── Genera token único en DB
   ├── Envía email con URL: /setup/[token]
   └── Operator completa: nombre, password, etc.

Tenant Admin crea Subordinate:
   ├── Genera token único en DB
   ├── Envía email con URL: /setup/[token]
   └── Subordinate completa: nombre, password, etc.

/setup/[token] flow:
   ├── Validar token en DB (no expirado, no usado)
   ├── POST /invitations/:token/complete
   ├── Completar perfil (nombre, password)
   ├── Crear/actualizar user en Supabase Auth
   ├── Asignar rol correspondiente
   └── Redirigir según rol
```

### Auth API Endpoints

| Endpoint | Método | Descripción |
|----------|--------|-------------|
| POST /auth/login | Login usuario | Returns tokens + user + dashboard_route |
| POST /auth/signup | Registro cliente | Crea user + org en Supabase |
| POST /auth/forgot-password | Solicitar reset | Envía email con link de recuperación |
| POST /auth/reset-password | Reset password | Actualiza password (requiere token del email) |
| POST /auth/dev-login | Login dev | Para desarrollo (superadmin) |
| POST /invitations | Crear invitación | Genera token para Operator/Subordinate |
| GET /invitations/:token | Validar token | Verifica si el token es válido |
| POST /invitations/:token/complete | Completar setup | Finaliza registro desde invitación |

### Session Management (Frontend)

```typescript
// 1. Después de login/signup exitoso:
localStorage.setItem('sb-access-token', data.access_token)
localStorage.setItem('sb-refresh-token', data.refresh_token)
localStorage.setItem('sb-user', JSON.stringify(data.user))

// 2. Para middleware ( Next.js ):
document.cookie = `sb-access-token=${data.access_token}; path=/; max-age=${expires_in}; SameSite=Lax`
document.cookie = `sb-refresh-token=${data.refresh_token}; path=/; max-age=${expires_in}; SameSite=Lax`

// 3. Para logout:
localStorage.removeItem('sb-access-token')
localStorage.removeItem('sb-refresh-token')
localStorage.removeItem('sb-user')
document.cookie = 'sb-access-token=; path=/; max-age=0'
document.cookie = 'sb-refresh-token=; path=/; max-age=0'
```

| patients:read | Ver pacientes |

| users:manage | Gestionar usuarios del tenant |
| roles:manage | Crear/editar roles |
| billing:manage | Ver/gestionar suscripción |
| analytics:view | Ver dashboard de analytics |
| settings:manage | Configuración del tenant |

### Session Management

- Clerk handles session storage, JWT validation
- Organization ID stored in Clerk publicMetadata
- Role + permissions fetched from DB on each request
- Redis caching para permissions (TTL: 5 min)
- MFA obligatorio para SaaS admins, opcional para tenants

- **AI:** API → AI Core (fallback chain: OpenAI → Anthropic → cached) → Redis cache

### i18n System
- Custom `useI18n()` hook in `@/lib/i18n`
- Instant language switch (React state, NO reload)
- Translations stored in `@/lib/i18n/translations.ts`

### Key Patterns
- Client Components with "use client" directive for interactivity
- React Query for client-side mutations and caching
- RLS in Supabase for tenant data isolation
- Audit logs for all write operations
- Feature flags gated by subscription plan
- **Auth:** Fetch to API + localStorage for user data + cookies for middleware

### Auth Session Pattern

```typescript
// Login response includes:
{
  access_token: string,
  refresh_token: string,
  expires_in: number,
  user: { id, email, role, org_slug, first_name, last_name },
  dashboard_route: string  // "/admin" or "/{slug}/dashboard"
}

// Store in localStorage:
localStorage.setItem('sb-access-token', data.access_token)
localStorage.setItem('sb-refresh-token', data.refresh_token)
localStorage.setItem('sb-user', JSON.stringify(data.user))

// Store in cookies (for middleware):
document.cookie = `sb-access-token=${token}; path=/; max-age=${expires_in}; SameSite=Lax`
```

## Code Organization Rules

4. **Server Components by default.** Only add "use client" when the component needs interactivity.
5. **Colocate related files.** Keep page-specific components next to their page.
6. **Modules in NestJS:** One module per feature, controllers/services/dto separate.
7. **Prisma:** One model per line in schema, use enums for status fields.

## 16. Reglas No Negociables

1. **Arquitectura multitenant.** Cada query DEBE filtrar por organizationId. Violar esto compromete la seguridad de datos médicos.
2. **TypeScript strict.** No `any`. Cada variable debe tener tipo conocido.
3. **Auditoría immutable.** Todas las operaciones de escritura (create, update, delete) deben generar AuditLog.
4. **Manejo de errores.** Cada función async en try/catch. Errores nunca deben propagarse silenciosamente.
5. **No .env en git.** Usar .env.example como plantilla.
6. **Mobile-first.** Diseñar para móvil, mejorar para desktop.
7. **Accesibilidad.** Todos los elementos interactivos accesibles por teclado, labels ARIA apropiados.
8. **AI Core fallback.** Si el agente primario falla, DEBE intentar el siguiente. Nunca retornar error al usuario por falla de IA.
9. **Feature gates por suscripción.** Verificar features del plan ANTES de permitir acciones, no después.
10. **GDPR/CCPA readiness.** No guardar datos personales sin consentimiento documentado.
11. **Testing obligatorio.** No se hace PR hasta que existan scripts de test (unit + integration) y estos pasen en CI. Coverage mínimo: 80%.
12. **PRs atómicos por feature/fase.** Un PR = una funcionalidad completa lista para producción. NO hacer PRs de "improve X" o "fix Y" — cada change debe ser parte de una feature o fase del build order. Los fixes menores se agrupan en el PR de la feature más cercana.
